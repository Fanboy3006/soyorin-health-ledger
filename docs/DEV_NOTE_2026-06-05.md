### 同步系统修复详情

#### 1. `synced` 字段更新修复
- **问题**：数据成功插入 Supabase 后，本地 IndexedDB 中的 `synced` 字段仍然是 `false`
- **原因**：`Dexie.Table.update()` 返回 0 时静默失败（记录可能已被修改）
- **修复**：检查 `update()` 返回值，为 0 时使用 `put()` 作为 fallback 直接覆盖整条记录

#### 2. `pullTable` 拉取逻辑修复
- **问题**：同步日志中只有 `pushTable`，没有 `pullTable`
- **原因**：`query.order('created_at')` 在 `daily_summaries` 表上失败（该表没有 `created_at` 列），错误被静默吞掉
- **修复**：检测 `created_at` 相关错误，自动重试不带 `order` 的查询；提取 `processPullData` 函数减少重复代码

#### 3. `presetId` 类型映射
- **问题**：第一台电脑预设 ID 是数字（1,2,3...），第二台电脑从云端拉取后无法关联
- **Push 方向**：推送 `ledger_entries` 时，将 `presetId` 从本地数字解析为 UUID（通过 `db.presetAssets.get(id).remoteId`）
- **Pull 方向**：拉取 `ledger_entries` 时，将 `presetId` 从 UUID 解析为本地数字（通过 `db.presetAssets.where('remoteId').equals(uuid).id`）
- **边界处理**：预设未找到时设为 `null`，预设未同步时保留数字等待下次同步

#### 4. `daily_summaries` 主键修复
- **问题**：`daily_summaries` 表使用 `date` 作为主键，但同步代码硬编码使用 `id`
- **修复**：引入 `TABLE_PK` 配置映射，所有 CRUD 操作动态使用正确的 PK 列

## 五、开发思想备忘

> **原则**：同步系统是离线优先架构的命脉，必须保证 Push/Pull 双向数据一致性。

**关键决策**：
1. **presetId 映射方案** — 选择在同步层做双向映射（Push 数字→UUID，Pull UUID→数字），而非修改数据库类型。优点是无需迁移现有数据，缺点是同步层复杂度增加。
2. **`TABLE_PK` 配置** — 将表主键信息集中管理，避免硬编码。未来新增表时只需在 `TABLE_PK` 中添加一行。
3. **`put()` fallback** — 当 `update()` 返回 0 时使用 `put()` 覆盖，确保 `synced` 状态一定能更新。这是防御性编程，防止 Dexie 内部状态不一致。

**技术债务提醒**：
- `sync.ts` 当前 505 行，随着功能增加可能继续膨胀。建议在下次大重构时拆分为 `push.ts` / `pull.ts` / `delete.ts`
- `toRemoteRow` 和 `toLocalRecord` 中的 switch-case 模式重复，可考虑使用配置对象 + 自动映射
- 同步错误处理目前只记录日志，没有用户可见的提示。建议添加 Toast 通知
